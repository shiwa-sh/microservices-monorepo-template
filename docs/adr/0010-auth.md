# ADR-0010: Authentication & Authorization

- **Status:** Accepted
- **Date:** 2026-05-19
- **Deciders:** Platform team
- **Related:** [ADR-0001](0001-language-and-runtime.md), [ADR-0006](0006-temporal.md), [ADR-0009](0009-api-gateway.md)

## Context

The platform supports:

- **B2C users** — public sign-up, email/password, social login, MFA, recovery.
- **B2B organisations** — multi-tenant orgs, SSO (OIDC/SAML), per-org roles, SCIM provisioning.
- **Service-to-service identity** — internal calls trusted by network policy ([ADR-0003](0003-cluster-topology.md)); identity headers are forwarded, no per-call token.
- **Third-party API consumers** — external developers obtaining tokens to call our public APIs.

Constraints inherited from earlier ADRs:

- **Self-host only.** No Auth0/Clerk/WorkOS.
- **Per-service cost matters.** JVM-based stacks pay a footprint tax incompatible with [ADR-0001](0001-language-and-runtime.md).
- **The edge validates tokens; services decide permissions.** Set in [ADR-0009](0009-api-gateway.md).

Hard requirement on the UI:

- **The login UI is a custom Next.js implementation of a Figma design.** It is not the IdP's hosted pages, themed templates, or a forked vendor UI. The IdP must expose a headless, API-first identity flow that Next.js drives end-to-end.

Authorization shape:

- Day-one features include cases beyond flat RBAC (resource ownership across orgs, shared resources, role-per-org-per-resource).
- Temporal is available ([ADR-0006](0006-temporal.md)), which makes the dual-write between application DB and authz store a routine workflow pattern.

The decision splits into:

1. **Identity Provider** — login, registration, sessions, user tokens.
2. **OAuth2 / OIDC authorization server** — machine and third-party tokens.
3. **Authorization engine** — given an authenticated principal, may they perform an action.

## Decision drivers

1. **Headless IdP** — required by the Figma constraint.
2. **Polyglot-tolerant integration contracts** — auth is a *protocol* contract (token format, header names, validation rules), expressible in any language. Per [ADR-0001](0001-language-and-runtime.md)'s escape hatches.
3. **Files in the repo** — identity schemas, OAuth2 client definitions, authz schemas all version-controlled.
4. **Operational shape consistent with the rest of the platform** — Go binaries on Postgres, not JVM.

## Considered options

### Identity Provider

- **Ory Kratos** — fully headless self-service flows; identity schemas as JSON Schema files; YAML config; Go codebase; Apache 2.0. The reference implementation of the headless pattern.
- **Zitadel** — single Go binary, native multi-tenancy; custom login is supported but is the road less travelled, and hosted pages are the design centre.
- **Keycloak** — most mature OSS IdP; comprehensive feature set; JVM footprint, custom UI via Authentication Sessions API is a minority path, extension work in Java.
- **Authentik** — Python/Django + Celery + Redis; B2C self-service flows less mature than Kratos.

### OAuth2 / OIDC server

- **Ory Hydra** — Go, Postgres, OAuth2/OIDC server, headless; integrates with Kratos for user-consent flows.
- Roll our own — explicitly rejected ([ADR-0000](0000-platform-foundations.md): no "temporary" infrastructure).

### Authorization engine

- **SpiceDB** — Zanzibar-style ReBAC; schema as `.zed` file; `zed validate` runs assertions and validations in CI; `zed` CLI for debugging; gRPC + Protobuf with official SDKs for Go/TS/Python/Java; single Go binary, Postgres-backed.
- **OpenFGA** — clean SDK, approachable docs; testing tooling trails SpiceDB.
- **Ory Keto** — keeps the Ory stack but OPL tooling and community trail SpiceDB.
- **In-process RBAC per service** — diverges per service; "occasional complex case" already exists from day one (cross-org sharing, role-per-resource); migrating to a Zanzibar engine later means a per-feature data migration.

## Decisions

### Identity Provider: Ory Kratos

Kratos handles human authentication: registration, login, MFA, recovery, account settings, social login. Identity schemas as JSON Schema files at `infra/auth/kratos/identity-schemas/`. All other config in YAML at `infra/auth/kratos/`. Deployed via Helm at `infra/helm/platform/ory/`.

The Next.js login UI lives in `apps/frontend/src/app/(landing)/auth/` and drives Kratos self-service flows directly.

The Kratos session cookie is configured `SameSite=Lax` + `Secure` + `HttpOnly` in `infra/auth/kratos/`. This is the
first line of CSRF defence (it blocks the classic cross-site POST); Kratos's built-in anti-CSRF token covers the
self-service flows themselves, and the edge Origin check ([ADR-0009](0009-api-gateway.md)) backstops other
cookie-authenticated mutations. The browser-side CSP and Server-Actions CSRF handling are [ADR-0014](0014-frontend.md)'s.

### OAuth2 / OIDC server: Ory Hydra, when a public API exists

Hydra is the OAuth2/OIDC server, deployed when a project exposes a public API or external machine clients (the
`hydra_thirdparty` flag, [ADR-0009](0009-api-gateway.md)). It issues **authorization-code and client-credentials tokens
for third-party developers and external machines.** Those tokens are validated at the edge by Oathkeeper and converted
to the standard identity headers ([ADR-0009](0009-api-gateway.md)), so the internal request shape is identical whether
or not Hydra is deployed.

Hydra is **not** used for internal service-to-service calls — those carry no token and are gated by Cilium NetworkPolicy
([ADR-0003](0003-cluster-topology.md)). When deployed, Hydra is the one and only issuer of external tokens; no service
mints its own.

Hydra config lives at `infra/auth/hydra/`. Third-party OAuth2 clients are declared in YAML and applied by a Hydra CLI
invocation in a Helm post-sync hook.

### B2B Organisations: an `orgs` service on top of Kratos

Kratos stores identities, not orgs. Multi-tenancy is implemented as a first-party `services/orgs/` service that owns:

- Organisations and memberships.
- The "active organisation" for a user session (set by a post-login Kratos webhook).
- Org-level invitations and SSO connection records.

The `orgs` service is built ahead of the first B2B feature.

### Authorization Engine: SpiceDB

SpiceDB is the authorization engine, accessed through a shared `authz.Checker` interface in `libs/go/authz/`. Services never call the SpiceDB SDK directly.

- Schema lives at `infra/auth/spicedb/schema.zed`. **One global schema**, not per-service, because authz relationships routinely cross service boundaries.
- Test assertions and validations live at `infra/auth/spicedb/tests/*.yaml`. CI runs `zed validate` on every PR.
- Deployed via Helm at `infra/helm/platform/spicedb/`, sharing the platform Postgres.
- **Deployed directly (Deployment + Service), not via the SpiceDB operator.** Authzed recommends the operator for production, but it ships only as a ~4000-line `bundle.yaml` (no official Helm chart) and its main value is automating datastore migrations. We instead run a `spicedb migrate head` initContainer before `serve` — the same pattern our own services use for their migrations. The trade-off: the rare SpiceDB release that needs a *phased* datastore migration is handled manually per that release's upgrade notes (intermediate `spicedb migrate <phase>` → roll the Deployment → `migrate head`). For normal install and minor/patch upgrades, the initContainer is fully automatic.

### Dual-write discipline

Every authz-relevant mutation (create / share / transfer / delete on a protected resource) is wrapped in a Temporal workflow ([ADR-0006](0006-temporal.md)). The workflow contains both the application-DB write and the SpiceDB write as activities. SpiceDB's `ZedToken` is propagated to subsequent reads where read-after-write consistency matters.

Direct application-DB writes that mutate authz-relevant tables outside a workflow are a review-blocker enforced by a `depguard`-style lint rule against the relevant store methods.

### Identity: validated at the edge, carried as headers

Tokens are validated once, at the edge ([ADR-0009](0009-api-gateway.md)). Past the edge there are **no tokens** —
identity travels as a fixed set of trusted headers, the same shape for every request.

- **Edge validation.** Ory Oathkeeper validates the Kratos session cookie (browser) or the Hydra-issued JWT
  (third-party / machine clients). JWTs are RS256 with keys at a JWKS endpoint; validation requires issuer, audience,
  expiry, signature, and `nbf`, with 30 s clock-skew tolerance, defined once in `docs/auth/jwt-validation.md`.
- **Identity headers.** Oathkeeper strips any client-supplied identity headers and injects authoritative `X-User-Id`,
  `X-Org-Id`, `X-Roles` ([ADR-0009](0009-api-gateway.md)). The underlying claim set is `sub`, `org_id` (active
  organisation), `roles` (org-scoped), plus standard timing claims.
- **Services read headers.** The Go reference reader is `libs/go/authmw/`; it reads the identity headers into a typed
  principal. Services never fetch JWKS or parse a JWT. Escape-hatch languages implement the same header contract.
- **Service-to-service calls** carry no token. They forward the same identity headers and are gated by Cilium
  NetworkPolicy ([ADR-0003](0003-cluster-topology.md)); only sanctioned callers can reach a service's port. This is the
  platform's internal trust boundary — network identity, not a per-call machine token.

A small **conformance test suite** at `tools/auth-conformance/` ships with the repo: identity-header fixtures with
expected principals and authorization outcomes. Any non-Go service must pass it.

### Day-one component cost (summary)

The internal-only auth stack at launch is:

- **Kratos** (Helm) — human login.
- **Oathkeeper** (Helm) — edge validation + identity-header injection ([ADR-0009](0009-api-gateway.md)).
- **SpiceDB** (Helm) — authorization.
- **`services/orgs/`** (first-party service).
- **`libs/go/authmw/`** (header reader) + **`libs/go/authz/`** (`Checker`) libraries.

**Hydra is deployed only when a project exposes a public API or external machine clients** (`hydra_thirdparty` flag,
[ADR-0009](0009-api-gateway.md)); internal-only projects do not run it. There is no service-to-service token: internal
trust is Cilium NetworkPolicy ([ADR-0003](0003-cluster-topology.md)). All components share the platform Postgres and the
platform observability stack.

## Consequences

### Positive

- Login UI freedom. The Figma design is implemented exactly, with no fight against vendor templates.
- One way to obtain a token for human or machine identity, day one and forever.
- Authz is a single coherent model from day one — no deferred migration, no per-service RBAC tables.
- Tooling supports the work: schema in files, tests in files, CLI for debugging, GitOps for everything.
- Polyglot-safe. Auth integration is a protocol contract; Rust or Python services participate on equal terms.
- Dual-write risk is structurally contained by routing authz-relevant mutations through Temporal.

### Negative / Risks

- **Three new platform components on day one for internal-only projects** (Kratos, Oathkeeper, SpiceDB) plus the `orgs` service; Hydra is added only when a public API exists. Accepted; each is best-in-class for its job, and the constraints (headless login from day one, ReBAC from day one) make consolidation impossible without compromise. Dropping the service-to-service token in favour of Cilium NetworkPolicy removes a per-call validation path from every service.
- **Building B2B organisations on top of Kratos** is real work. Accepted; the alternative (a bundled-orgs IdP) costs the Figma requirement.
- **Dual-write discipline must be enforced.** A direct DB write that skips the Temporal workflow is a silent authz bug. Mitigated by lint rules, code-review checklist, and integration tests asserting SpiceDB state after every workflow.

### Follow-ups

- `infra/helm/platform/ory/` (Kratos + Oathkeeper, plus Hydra when `hydra_thirdparty` is set) and `infra/helm/platform/spicedb/`.
- `infra/auth/oathkeeper/` access rules (validate session/JWT → strip → inject identity headers).
- `libs/go/authmw/` (JWT middleware) and `libs/go/authz/` (`Checker` interface + SpiceDB-backed implementation).
- `infra/auth/kratos/`, `infra/auth/hydra/`, `infra/auth/spicedb/`.
- `services/orgs/` scaffold.
- `tools/auth-conformance/` conformance suite.
- `docs/auth/jwt-validation.md` rules.
- `docs/auth/dual-write.md` documenting the Temporal pattern for authz-relevant mutations.

## Rules

- Human identity is owned by Ory Kratos. There is no alternate user store.
- External tokens (third-party and machine clients) are issued by Ory Hydra, which is deployed only for projects exposing a public API (`hydra_thirdparty` flag, [ADR-0009](0009-api-gateway.md)). No service issues its own tokens.
- The login UI is the Next.js app driving Kratos self-service flows. Kratos hosted pages are not used.
- The Kratos session cookie is `SameSite=Lax` + `Secure` + `HttpOnly`. CSRF defence is layered: SameSite cookies, Kratos's built-in token on auth flows, Next.js Server-Actions Origin checks ([ADR-0014](0014-frontend.md)), and the edge Origin check on other cookie-authenticated mutations ([ADR-0009](0009-api-gateway.md)).
- B2B organisations are owned by `services/orgs/`. Other services consult it via its HTTP API.
- Authorization is SpiceDB, accessed only through `libs/go/authz/`'s `Checker` interface. Direct SpiceDB SDK use outside this library is forbidden.
- The SpiceDB schema is a single global file at `infra/auth/spicedb/schema.zed`. Per-service schemas are forbidden.
- Authz-relevant mutations run inside a Temporal workflow with the DB write and the SpiceDB write as separate activities. Direct DB writes to authz-relevant tables outside a workflow are forbidden.
- Inline role checks in handlers are forbidden. Every permission decision goes through `Checker.Allowed(ctx, action, resource)`.
- Tokens are validated once at the edge by Oathkeeper ([ADR-0009](0009-api-gateway.md)); services do not validate tokens. JWTs are RS256 with keys at JWKS; validation rules are defined in `docs/auth/jwt-validation.md`.
- Identity is carried platform-wide as `X-User-Id`, `X-Org-Id`, `X-Roles` headers, injected by Oathkeeper at the edge and forwarded unchanged on internal calls. Services read identity only from these headers and never parse a token. Renaming the headers requires editing [ADR-0009](0009-api-gateway.md) and this ADR.
- Service-to-service calls carry no token; they forward the identity headers and are gated by Cilium NetworkPolicy ([ADR-0003](0003-cluster-topology.md)). Shared secrets, HMAC schemes, and per-call machine tokens are forbidden.
- Non-Go services read the identity headers via the same contract and pass `tools/auth-conformance/` before merging.
