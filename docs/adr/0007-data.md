# ADR-0007: Data & Migrations

- **Status:** Accepted
- **Date:** 2026-05-19
- **Deciders:** Platform team
- **Related:** [ADR-0001](0001-language-and-runtime.md), [ADR-0003](0003-cluster-topology.md), [ADR-0006](0006-temporal.md)

## Context

Every backend service needs persistent state. We need a single answer to: engine, operation, schema evolution, service access, and local–prod parity.

Several platform components already require Postgres: Kratos/Hydra ([ADR-0010](0010-auth.md)), SpiceDB ([ADR-0010](0010-auth.md)), Temporal ([ADR-0006](0006-temporal.md)). Adding a second engine for application services would mean operating two engines.

Authz-relevant mutations are workflows ([ADR-0006](0006-temporal.md), [ADR-0010](0010-auth.md)). The database does not need a built-in outbox.

## Decision drivers

1. **One engine.** Polyglot persistence is a permanent operational tax at 100-service scale.
2. **Service-owned data.** A service's data is private; other services read it only through that service's HTTP API.
3. **Schemas as code in the repo.** Migrations are SQL files in the service folder, code-reviewed.
4. **Local–prod parity.** Whatever produces a working DB in production must work locally with one command.
5. **No extensions become tribal knowledge.** Extensions (PostGIS, TimescaleDB, pgvector) are per-service decisions recorded in the service README.

## Decisions

### Engine: PostgreSQL

Postgres is the engine for every service. The cluster runs the current stable major version; version pin lives in `infra/helm/platform/postgres/`.

- Already required by Kratos/Hydra, SpiceDB, and Temporal.
- Extension ecosystem covers anticipated needs without changing engine.
- Strong consistency, transactional DDL, mature replication, mature backup tooling.
- Best-in-class Go driver (`pgx`) with first-class sqlc support.

### Topology: database per service, shared cluster

Each service owns **one logical database** on a shared Postgres cluster.

- Service A cannot connect to service B's database. Credentials are per-service.
- No cross-database queries. Cross-service data needs are satisfied via HTTP / OpenAPI ([ADR-0008](0008-api-contracts.md)).
- Shared physical cluster keeps operational cost manageable. Read replicas are introduced per [ADR-0003](0003-cluster-topology.md) when single-service load justifies them.
- Services with compliance / isolation requirements get their own cluster via a service-specific ADR — never their own schema in the shared cluster.

### Cluster operator: CloudNativePG (CNPG)

CNPG runs the Postgres cluster.

- Kubernetes-native CRDs (`Cluster`, `Pooler`, `Backup`, `ScheduledBackup`).
- HA via streaming replication; failover and switchover as CRD operations.
- Built-in `Pooler` CRD deploys PgBouncer alongside the cluster.
- Backups to S3-compatible object storage (the external bucket from [ADR-0003](0003-cluster-topology.md)) with PITR via Barman.
- Apache 2.0; CNCF Sandbox.

Considered Zalando, Crunchy, StackGres, raw Patroni — CNPG wins on Kubernetes-nativity and single-binary operator.

### Major-version upgrades

Postgres major upgrades happen via CNPG's logical replication path: a new `Cluster` is created at the new version; CNPG replicates from the old; traffic cuts over with a brief read-only window per service. Documented in `docs/database/major-upgrade.md`. Performed once per Postgres release cadence (~yearly) during a planned maintenance window.

### Connection pooling: PgBouncer in transaction mode

PgBouncer sits between every service and the cluster, deployed via CNPG's `Pooler` CRD.

- **Transaction mode** is the default. Session-mode features (advisory locks, `SET LOCAL`, `LISTEN/NOTIFY` across statements, server-side prepared statements without compatibility flags) are **not permitted in application code**. CI checks against representative patterns.
- The `pgx` driver is configured for PgBouncer compatibility (`statement_cache_mode = describe`).
- Per-service connection limits are sized in PgBouncer config, not in `pgx`. App-side pool is small (10) per replica; PgBouncer multiplexes onto a smaller server-side pool.
- A second `Pooler` instance in **session mode** is provided cluster-wide for the rare workload that genuinely needs it (`LISTEN/NOTIFY`-driven internal tooling). Using it requires a one-paragraph note in the service README.

### SQL access: sqlc + pgx, no ORM

Services access their database through **sqlc-generated Go code** over `pgx`.

- SQL queries live in `services/<service>/internal/store/queries/*.sql`.
- Generated Go in `services/<service>/internal/store/`.
- `sqlc.yaml` configures the Postgres engine and pgx driver.

ORM abstractions hide query plans, which becomes a liability at scale. sqlc gives typed Go from real SQL with no runtime cost.

### SQL linting: sqruff

`sqruff` (a Rust, single-binary, sqlfluff-compatible linter) lints both queries and migrations. It reads the
sqlfluff-format config in `.sqlfluff` at repo root with `dialect = postgres`. Distributed as a prebuilt binary (no
Python toolchain), it runs via lefthook pre-commit and CI.

### Migrations: dbmate

`dbmate` is the migration tool.

- Files live at `services/<service>/migrations/<YYYYMMDDHHMMSS>_<description>.sql`.
- Each migration is one SQL file with `-- migrate:up` and `-- migrate:down` sections.
- **Down migrations are not used in production.** Production rollback is a forward fix — a new migration that reverses the change. Down sections exist for local development only.

### Where migrations run

**Production:** `dbmate up` runs as an **init container** before the service container in the shared service Helm chart ([ADR-0004](0004-gitops.md)). The service starts only after the init container succeeds. `dbmate` uses an advisory lock to prevent concurrent runs across replicas.

**Local:** the same `dbmate` binary against the local database.

```sh
mise run db:up          # local Postgres + PgBouncer
mise run db:migrate     # dbmate up for every service
mise run db:seed        # optional fixtures
mise run -C services/<svc> db:migrate   # service-scoped
```

The init container image is the same one engineers run locally. No separate migration image that drifts from production.

### Zero-downtime schema changes: expand → migrate → contract

All schema changes that could break running code follow expand/contract:

1. **Expand.** Add new columns/tables, nullable or defaulted. Deploy.
2. **Migrate data.** Backfills run as **Temporal workflows** ([ADR-0006](0006-temporal.md)), not as SQL scripts. Backfills are observable, resumable, and rate-limitable. Deploy if app code needs new behaviour.
3. **Contract.** Drop old columns/tables only after no code references them.

Hard rules:

- No destructive migration runs in the same deploy as code that uses the new schema. At least one deploy boundary between expand and code switch.
- No `ALTER TABLE` that takes a long lock without `lock_timeout` and a documented online-DDL plan. Long-lock migrations are blocked at review.
- No `DROP COLUMN` until application code has shipped without referencing it for at least one full prod deploy cycle.

### Multi-tenancy: row-level

For B2B organisations, tenant isolation is row-level: every multi-tenant table has an `org_id` column. Service code filters by `org_id` from the request context.

- **Postgres Row-Level Security (RLS)** is enabled on multi-tenant tables. `CURRENT_SETTING('app.org_id')` is set per request via `pgx`'s `BeforeAcquire` hook.
- Schema-per-tenant and database-per-tenant patterns are not used. They explode the migration matrix.
- Tenants requiring physical isolation earn their own service (and possibly cluster), not their own schema.

### Extensions

- **Day-one cluster manifest pre-allows:** `pgcrypto`, `uuid-ossp`, `pg_stat_statements`. These are common enough to be cheap.
- **Sanctioned per-service when justified:** PostGIS (geospatial), TimescaleDB (time-series), pgvector (embeddings).
- **Adoption rule:** the first service that needs a sanctioned extension installs it via a migration running `CREATE EXTENSION IF NOT EXISTS …`. The service README's "Postgres extensions" section is updated. CNPG cluster manifests pre-allow the extension list.
- Other extensions require an ADR.

### Credentials

- Each service has a dedicated Postgres role with access only to its own database.
- Roles, grants, and the connection password live in a SOPS-encrypted file per [ADR-0005](0005-secrets.md); the sops-operator materialises them as Kubernetes Secrets that the service Helm chart mounts.
- The CI/test connection string is local-only and committed in `.env.example` with non-secret defaults.

### Backups and PITR

Covered by [ADR-0003](0003-cluster-topology.md)'s Backups section — CNPG writes to the off-cluster bucket with WAL-based
PITR, restore rehearsed quarterly. No service-specific exceptions.

## Consequences

### Positive

- One engine, one mental model.
- Service ownership is structural: a service cannot read another service's data even if it tried.
- Migrations live with the code they describe; the same diff carries behaviour and schema.
- Local matches production: same `dbmate`, same SQL, same init-container shape.
- Backfills as Temporal workflows make zero-downtime schema changes a normal review item.

### Negative / Risks

- PgBouncer transaction mode rules out a narrow set of Postgres features. Mitigated by lint rules and the session-mode `Pooler` escape hatch.
- Shared physical cluster means a noisy neighbour can affect all services. Mitigated by CNPG resource isolation, per-role slow-query alerting, and per-service connection limits.
- No ORM is unfamiliar to engineers coming from Django/Rails/Spring. Paid once during onboarding.
- Forward-fix as the only rollback path requires discipline. Documented in `docs/database/migrations.md`.

### Follow-ups

- `infra/helm/platform/postgres/` deploying the CNPG cluster, `Pooler`s, and `ScheduledBackup`.
- `services/_template/` skeleton including `migrations/`, `internal/store/`, `sqlc.yaml`, and a sample migration.
- Repo-level `.sqlfluff`.
- `mise` tasks: `db:up`, `db:migrate`, `db:seed`, `db:reset`, scoped per-service and globally.
- `docs/database/migrations.md` (expand/contract, locking discipline, forward-fix policy).
- `docs/database/multi-tenancy.md` (`app.org_id` session variable, RLS policies).
- `docs/database/major-upgrade.md` (logical-replication cutover procedure).
- Quarterly backup-restore rehearsal as a Temporal `Schedule`.

## Rules

- Every service uses Postgres. Other engines (MySQL, MongoDB, DynamoDB) are not permitted without their own ADR.
- Each service owns exactly one logical database. Credentials are scoped to it; cross-database queries are forbidden.
- Cross-service data access is HTTP via the generated OpenAPI client.
- The Postgres cluster is operated by CloudNativePG. Connection pooling is PgBouncer via CNPG `Pooler` in transaction mode.
- Application code must be transaction-mode safe: no advisory locks, no `SET LOCAL`, no cross-statement `LISTEN/NOTIFY`, no server-side prepared statements without compatibility flags.
- SQL access is sqlc + pgx. ORMs (GORM, ent, bun) are not used.
- Migration files live at `services/<service>/migrations/` and are managed by `dbmate`.
- Migrations run as the service's init container in production. The same `dbmate` binary runs locally via `mise run db:migrate`.
- Down migrations are local-only. Production rollback is forward-fix.
- Schema changes follow expand → migrate (Temporal workflow) → contract, with at least one prod deploy boundary between expand and code switch.
- Multi-tenant tables carry `org_id` and have RLS enabled keyed on `app.org_id` set per request.
- Sanctioned extensions are PostGIS, TimescaleDB, pgvector, plus the cluster-default set (`pgcrypto`, `uuid-ossp`, `pg_stat_statements`). Other extensions require an ADR.
- Backups go to the off-cluster bucket. Restore is rehearsed quarterly.
- Database credentials reach pods via the sops-operator from SOPS-encrypted files; ad-hoc `kubectl create secret` is forbidden.
